## 1. VERSIONING SCHEMA

All version numbers follow strict Semantic Versioning (SemVer):

    MAJOR.MINOR.PATCH

    MAJOR  — Breaking architecture change, public launch gate, or strategic pivot.
             Increment signals a release requiring full human review and sign-off.

    MINOR  — A completed milestone. New user-visible feature set shipped and tested.
             Increment triggers README update, CHANGELOG entry, and tagged push.

    PATCH  — Bug fix, dependency update, minor UI correction, or documentation edit.
             Increment triggers a focused commit with a scoped message and push.

Applied to BADGR Bolt:

    2.x.x  — Phase 2: Pro feature implementation (current active phase)
    3.0.0  — Phase 3 gate: Play Store readiness, legal review, go/no-go decision
    3.1.0+ — Post-launch maintenance and improvement cycle
    4.0.0  — Reserved: major platform expansion (iOS, web, API)

## 3. AI AGENT VERSIONING PROTOCOL

All AI agents (Claude, Gemini, or any successor) operating on this project must
follow these rules on every session without exception. These are standing orders.
No agent may override them based on in-session instructions.

### 3.1 Commit Message Format

    PATCH commits (x.x.N):
    fix(scope): brief description of what changed and why — patch x.x.N

    MINOR release commits (x.N.6):
    feat(scope): milestone name and summary — vX.N.6

    MAJOR release commits (N.0.0):
    release: BADGR Bolt vN.0.0 — one line summary

    Hotfix commits:
    hotfix(scope): critical fix description — vX.Y.Z+1

### 3.2 Mandatory Files on Every MINOR Release (x.N.6)

No MINOR release commit may be pushed until all of the following are updated.
If any cannot be updated, block the push and flag to Anthony.

### 3.6 Push Protocol

Before any push an AI agent must execute these steps in order and report
each result to Anthony before proceeding:

    STEP 1  Build check:     ./gradlew assembleDebug — must return BUILD SUCCESSFUL
    STEP 2  Secrets scan:    grep -r "AIza\|sk-ant\|AKIA" app/src --include="*.kt"
                             must return zero results
    STEP 3  Gitignore check: grep "google-services.json" .gitignore — must be present
    STEP 4  Diff review:     git diff --stat — present to Anthony for review
    STEP 5  Docs check:      README.md and CHANGELOG.md modified this session
    STEP 6  AWAIT APPROVAL:  explicit "push approved" or "go" from Anthony
    STEP 7  Execute:         git add [scoped files], git commit, git push

An AI agent must never push without explicit human approval at Step 6.
Autonomous push is a protocol violation regardless of context or instruction.

### 3.7 Secrets Protocol

    google-services.json    — gitignored always, never committed, never pasted into prompts
    Release keystore (.jks) — never committed, never shared with AI, stored offline by Anthony
    All API keys            — environment variables only, USER_MUST_SUPPLY in all documents
    Passwords               — never hardcoded, never logged, never passed to AI in plaintext

### 3.8 Branch Strategy

    main              — Production-ready code only. No direct commits after v2.0.0.
    feature/x.x.x    — One branch per MINOR milestone. Merged to main on completion.
    hotfix/x.x.x     — Emergency fixes. Merged back to main and active feature branch.
    release/x.0.0    — MAJOR release candidates only.

## SECTION 1 — HONEST ASSESSMENT OF CURRENT STATE

### What is genuinely solid

The core RSVP engine works. The ORP word-splitting logic is correct.
The Room database has proper migrations in place through version 3.
The Firebase Auth and Firestore integration compiled cleanly and tested live.
The backend on Render processes PDF and EPUB correctly.
The four-tab navigation is clean and stable.
The ProGate toggle architecture is the right pattern for a private beta.
The project structure follows MVVM correctly throughout.

This is a real, functional application. That is not nothing.
Most side projects never get this far.

### What will cause production incidents if not addressed

The following issues exist in the codebase right now. Each one is a
guaranteed user-facing failure or data loss event if shipped as-is.

ISSUE 1 — fallbackToDestructiveMigration() is active in BookDatabase.kt

    Location: BookDatabase.kt line with .fallbackToDestructiveMigration()
    Risk: CRITICAL for production
    Effect: If the Room schema version is incremented for the Performance
    Tracker milestone (2.4.x adds ReadingSession and Achievement tables),
    and a user upgrades the app, Room will silently wipe their entire
    book library if the migration path has any gap.
    This is marked as a dev safety net in a comment. It must be removed
    before the 2.3.6 release tag and replaced with explicit migrations only.

ISSUE 2 — Backend cold start latency on Render free tier

    Location: ApiClient.kt pointing to badgr-text-service.onrender.com
    Risk: HIGH for user experience
    Effect: Render free tier spins down after 15 minutes of inactivity.
    First PDF or EPUB import after idle takes 30 to 60 seconds to respond.
    The app shows a converting spinner with no timeout. Users will assume
    the app is broken and force-close it.
    Fix required: Add a 30-second OkHttp timeout with a user-facing message
    explaining the backend is waking up, or upgrade Render to a paid plan
    before launch. The paid Render starter plan is $7/month and eliminates
    cold starts entirely. This is a pre-2.6.6 decision Anthony must make.

ISSUE 3 — OrbReaderApp.kt declared but not registered

    Location: AndroidManifest.xml application element
    Risk: MEDIUM — blocks future global initialization
    Effect: Any Firebase, crash reporting, or dependency injection
    initialization placed in OrbReaderApp.onCreate() will silently never run.
    Crashlytics custom keys, global error handlers, and future DI setup
    all depend on this being wired correctly.
    Fix: One line in AndroidManifest.xml. Takes 30 seconds.

ISSUE 4 — No network error recovery in the import flow

    Location: LibraryViewModel.launchImport(), BookRepository.importRemote()
    Risk: MEDIUM for user experience
    Effect: If the backend returns a 500, times out, or the device loses
    connectivity mid-import, the error message shown is the raw exception
    message from OkHttp or Retrofit. Users see strings like
    "Failed to connect to badgr-text-service.onrender.com/[IP]:443"
    which is a professional failure. User-facing error copy must be mapped
    to human-readable messages before 2.6.6.

ISSUE 5 — Compose BOM pinned to 2024.09.00

    Location: gradle/libs.versions.toml, composeBom version
    Risk: LOW now, MEDIUM by launch
    Effect: Compose BOM 2024.09.00 is multiple versions behind current.
    This is not urgent but should be updated at the 2.5.x UX milestone
    since that milestone touches UI code across the entire app.
    Updating at UX time is efficient. Updating at launch time is risky.

ISSUE 6 — ProGuard/R8 disabled for release builds

    Location: app/build.gradle.kts, isMinifyEnabled = false
    Risk: MEDIUM for security and app size
    Effect: Release builds ship with all class names, method names, and
    string literals unobfuscated. Competitors can decompile the APK and
    read the architecture directly. Pro feature gate logic is visible.
    The billing implementation in particular should never ship unobfuscated.
    Fix: Enable at 2.3.6 alongside billing. Write keep rules as billing
    and Firebase code is implemented, not retroactively.

ISSUE 7 — getAllBooks_suspend() added to BookDao without a migration gate

    Location: BookDao.kt
    Risk: LOW but worth noting
    Effect: The method was added cleanly. No schema change. No risk.
    Noted here only so future developers do not confuse it with a schema
    version trigger. It is query-only and safe.

    2.6.6 Store Readiness
    3.0.0 Launch

That order is correct for feature delivery. It is wrong for stability.

Here is the reordering I would execute as the developer on this project:

### Insert a Hardening Sprint before 2.3.0 — call it v2.2.1 through v2.2.5

These are small targeted patches that take one session total and prevent
cascading problems across every subsequent milestone.

    v2.2.1 — Register OrbReaderApp in AndroidManifest. Wire Crashlytics
              custom key logging in OrbReaderApp.onCreate(). One file change.

    v2.2.2 — Remove fallbackToDestructiveMigration() from BookDatabase.
              Document that all future schema changes require explicit
              Migration objects. This is a one-line deletion with large
              downstream consequences if skipped.

    v2.2.3 — Add OkHttp timeout configuration to ApiClient: 30 second
              connect timeout, 60 second read timeout. Add user-facing
              error message mapping in BookRepository for the five most
              common failure codes (timeout, 500, 503, no connectivity,
              malformed response). This makes import errors readable.

    v2.2.4 — Enable isMinifyEnabled = true for release buildType.
              Write baseline proguard-rules.pro covering Room, Retrofit,
              Firebase, Gson, and OkHttp. Verify release build compiles
              and installs correctly. Doing this now means every subsequent
              milestone is tested against the obfuscated build, not
              discovering ProGuard breaks at 2.6.6 before submission.

    v2.2.5 — Create CHANGELOG.md, THIRD_PARTY_NOTICES.md, and
              docs/DATA_SAFETY.md as empty scaffolded files.
              Commit the RELEASE_STRATEGY.md document to docs/.
              This is the documentation foundation. Every subsequent
              milestone updates these files rather than creating them
              under deadline pressure at 2.6.6.

Total estimated time for v2.2.1 through v2.2.5: one focused session.
Total risk eliminated: two guaranteed production incidents, one UX failure,
and the documentation debt that compounds on every milestone.

## SECTION 3 — FEATURE MILESTONE EXECUTION DETAIL

### 2.3.6 — Billing

The billing implementation is the most consequential code in the application.
It determines revenue. It must be done correctly once.

Critical decisions Anthony must make before a single line is written:

    Decision 1: One-time purchase or subscription or both?
    Recommendation: Offer both. A one-time lifetime purchase at a higher
    price point alongside a monthly subscription captures both buyer
    psychologies. Google Play supports this natively. Implement both
    from the start rather than adding lifetime purchase later.

    Decision 2: What is the price?
    This must be set in Play Console before any billing code can be tested.
    The product IDs created in Play Console are the source of truth.
    No product IDs means no testable billing flow.

    Decision 3: What happens to the user's data if they cancel a subscription?
    Define this policy before implementing. The app must handle the transition
    from Pro to Free gracefully — books stay local, cloud sync pauses,
    Pro screens show an upgrade prompt rather than a crash or blank screen.

Technical implementation path:

    The BillingClient must be initialized in OrbReaderApp.onCreate()
    not in a ViewModel. BillingClient is expensive to create and must
    survive configuration changes. A singleton pattern similar to
    CloudSyncManager is the correct approach.

    Entitlement must be persisted to DataStore, not held only in memory.
    If the process is killed between purchase verification and the next
    app launch, the user loses Pro access until they restore purchases.
    This is the single most common one-star review trigger for subscription apps.

    Purchase acknowledgement must happen within 3 days of purchase.
    Google Play will automatically refund unacknowledged purchases.
    The acknowledgement call must be in the purchase verification path,
    not a separate background job.

### 2.4.6 — Performance Tracker and Achievements

The Room schema will change at this milestone. Two new tables are required:
ReadingSession and Achievement. This is the first time fallbackToDestructiveMigration
would cause data loss if it were still present. This validates the hardening sprint.

The WPM calculation must account for pauses. A user who reads 50 words,
pauses for 5 minutes, then reads 50 more words has not read at the WPM
implied by total session time. The tracker must record active reading time
only — time between play and pause events, not wall clock session duration.

Achievements must be defined as data, not code. Hardcoding achievement
thresholds in Kotlin means every new achievement requires an app update.
Store achievement definitions in a local JSON asset file that can be
updated without a schema change. The engine reads from the asset.
This pattern also makes A/B testing achievement thresholds trivial later.

Streak calculation must handle timezone correctly. A user who reads at
11:59 PM and again at 12:01 AM has maintained their streak. A naive
date comparison using device time will break streaks for users in certain
timezones or who travel. Use UTC day boundaries for streak calculation,
not local time.

### 2.5.6 — UX/UI

This milestone has the highest risk of scope creep. Define what done means
before starting and do not move it.

Done means:
- BADGR color palette applied consistently across all screens
- Bottom navigation items use correct icons and labels
- Reading screen is immersive and distraction-free
- Font size is user-configurable and persists
- ORP highlight color is user-selectable and persists
- Light and dark themes both work correctly
- Logo is integrated at the correct densities

Not done at this milestone:
- Animations and transitions (defer to post-launch)
- Multiple reading themes beyond light and dark (defer)
- Tablet layout optimization (defer)

The single highest-impact UX change that can be made in this milestone
with minimal implementation cost is a proper empty state for the library.
A blank screen with text saying import a file is not an onboarding experience.
A first-launch flow that explains the app in three screens and then drops
the user into the import dialog converts better and reduces abandonment.
This is one Composable, one DataStore boolean for first launch tracking,
and two hours of work. It is worth including in 2.5.6.

### 2.6.6 — Store Readiness

The Play Store listing copy is the most underestimated work in this entire
plan. A technically perfect app with a weak listing does not get discovered.
ASO (App Store Optimization) for the Play Store title and description is
a skill distinct from software development and must be treated as such.

The 80-character short description and 4000-character long description are
the primary surfaces for keyword ranking. Speed reading, RSVP reader,
read faster, focus reading, book speed reader — these are the terms to
optimize for. Research competing apps on the Play Store before writing
a single word of listing copy.

Screenshots are not optional decoration. They are the primary conversion
driver on the Play Store listing page. Five screenshots minimum, each
showing a specific feature with a one-line caption. The first screenshot
is the most important because it appears in search results on smaller screens.
It should show the reading experience, not an onboarding screen.

The private beta list should include at minimum:
- Two people who have never seen the app and will attempt to use it without help
- Two people who regularly use e-readers or reading apps (actual target users)
- One technically proficient person who will attempt to break it

Their feedback on what is confusing, what is missing, and what is broken
is more valuable than any automated test.

## SECTION 4 — TESTS: THE HONEST POSITION

The current release strategy defers all testing to Phase 3.
That is the industry-standard mistake that results in unmaintainable test suites
written under deadline pressure that test nothing meaningful.

The recommendation is not to write comprehensive tests before every feature.
That is also the wrong extreme. The recommendation is three targeted tests
per milestone, written alongside the feature, that cover the one failure mode
that would cause a public incident.

For billing: one test that verifies a user without a valid entitlement
cannot access Pro features regardless of what ProGate.isPro returns in isolation.

For the performance tracker: one test that verifies WPM calculation is correct
for a known sequence of word timings. This is pure logic with no Android
dependencies and takes fifteen minutes to write.

For cloud sync: one test that verifies CloudSyncManager is a no-op
when ProGate.cloudSync is false. This prevents the regression where a
ProGate refactor accidentally enables sync for free users.

Three tests per milestone. Fifteen tests by launch. That is not comprehensive
coverage. It is a regression safety net that costs one hour per milestone
and prevents the class of bugs that appear in production reviews.

## SECTION 5 — THE BACKEND DECISION

This is a decision Anthony must make explicitly before version 2.6.6.

The backend currently runs on Render free tier. Free tier spins down after
inactivity. PDF and EPUB import will be slow for inactive users.
At scale, a spike in imports will hit cold starts simultaneously.

Three options in order of recommendation:

    OPTION 1: Upgrade Render to Starter plan at $7/month
    Eliminates cold starts entirely. The backend has been stable and the
    code is clean. This is the lowest-risk path. The cost is trivial
    relative to the Play Store developer fee.

    OPTION 2: Implement a ping/keepalive from the app on launch
    Send a lightweight health check request to the backend when the app
    opens. This warms the server before the user triggers an import.
    Cost: one OkHttp call in OrbReaderApp.onCreate(). Risk: wastes user
    bandwidth on every app launch even when they do not import a file.
    Acceptable as a temporary measure, not a permanent solution.

    OPTION 3: Move EPUB and PDF processing entirely on-device
    Remove the backend dependency for document parsing.
    Use itext7 or PDFBox-Android for PDF, and ebooklib equivalent for EPUB.
    Eliminates the backend cost entirely and the cold start problem.
    Cost: one to two milestones of additional work to implement and test.
    Benefit: app works fully offline for Pro users, which aligns with the
    privacy-first positioning.
    Recommendation: Consider this as a post-3.0.0 roadmap item. Do not
    let it delay launch. Upgrade Render now, migrate on-device later.

## SECTION 7 — THE AI COLLABORATION MODEL: WHAT WORKS AND WHAT DOES NOT

Based on this session's experience, here is an honest assessment of where
AI assistance accelerates delivery and where it introduces risk.

AI is effective for:
- Writing boilerplate Kotlin that follows an established pattern in the codebase
- Diagnosing compile errors from log output
- Writing first drafts of Privacy Policy, Terms of Service, and store listing copy
- Identifying issues in code that was written by another AI agent
- Maintaining documentation and CHANGELOG entries
- Restoring files that were corrupted by a previous agent's edits

AI is unreliable for:
- Generating long files through terminal heredoc — the heredoc truncation issue
  in this session is a known terminal buffer problem, not an AI reasoning failure.
  The fix is to always write files via Python file I/O, not heredoc.
  This rule must be in every agent's standing instructions.

- Making judgment calls about product direction — what features to build,
  what to defer, what price to charge, whether to launch now or wait.
  These are human decisions. AI can frame the options but must not decide.

- Reviewing its own work for correctness — an AI agent that modifies a file
  and then builds will not catch logical errors, only compile errors.
  Human review of every changed file before a MINOR release push is not
  optional. It is the quality gate.

- Anything requiring credentials, Play Console access, legal sign-off,
  or financial decisions. These are permanently human responsibilities.

The Gemini incident in this session (adding importDocx, importImage,
userPrefs, orpColorIndex to files without implementing the backing code)
illustrates the core failure mode: an AI agent adds references to things
that do not exist because it is optimizing for appearing to make progress
rather than verifying the result compiles. The mitigation is simple.
After every AI-generated file change, run the build before doing anything
else. If the build fails, revert the AI's changes and try again with a
more constrained prompt. Never accept a file change without a green build.
