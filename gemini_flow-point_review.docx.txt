Phase 4 — Maintenance and Continuous Improvement
Objective: Establish a repeatable, safe loop for bug fixes, minor features, and performance improvements, keeping AI assistance useful without introducing unreviewed risk.

4.1  Bug and Feature Request Logging
    69. Create docs/BUGS.md and docs/FEATURE_REQUESTS.md in the repo.
    70. When a bug is reported (by user review, crash report, or internal test), log it with: description, reproduction steps, device/OS version, severity (Critical/Major/Minor).
    71. When a feature request is logged, note: source (user review, internal), priority, alignment with open-core strategy.
    72. Review both files weekly. Promote items to docs/BACKLOG.md when ready to implement.

4.2  Turning Bugs into AI Tasks
Rule: Never ask an AI to fix a bug it cannot reproduce from the information given.
Always include: the error message or stack trace, the file and function name,
the expected behavior, and the actual behavior.
Always ask for a test that would catch this bug if it reappeared.

PERSONA: You are an expert Android engineer debugging a Kotlin Android app.

CONTEXT: BADGR Bolt — speed-reading app. File: [FILENAME]. Function: [FUNCTION_NAME].

BUG REPORT:
  Description: [DESCRIBE THE BUG]
  Stack trace or error message: [PASTE STACK TRACE]
  Expected behavior: [WHAT SHOULD HAPPEN]
  Actual behavior: [WHAT ACTUALLY HAPPENS]
  Device: [DEVICE MODEL], Android [VERSION]

TASK: Identify the root cause and propose a fix.

REQUIREMENTS:
  - Do not modify files outside [FILENAME] unless strictly necessary.
  - Propose a unit test that would catch this regression.
  - If the fix requires a dependency change, list it explicitly.
  - If the root cause is ambiguous, list the top 2 hypotheses and how to test each.

OUTPUT: Root cause analysis, proposed fix (code), and regression test.

4.3  Safe AI-Assisted Refactoring
    73. Refactors must only be proposed after tests exist for the code being changed. No tests = no refactor via AI.
    74. Scope each refactor to one file or one logical unit. Never refactor the entire codebase in one step.
    75. After AI proposes a refactor, the human developer runs the test suite and reviews the diff before committing.
    76. Common safe refactors: extract duplicated logic to a utility function, improve coroutine scope management, reduce memory allocations in the RSVP render loop.

4.4  Performance Improvement Loop
    • Large file memory: if importing a 500-page PDF causes an OOM error, chunk parsing into pages and process them lazily. Use this as a prompt task for the AI.
    • RSVP timing drift: profile with Android Profiler. If timing drifts beyond 5ms per word, investigate coroutine scheduling and consider using a Handler with a fixed delay on a dedicated thread.
    • Startup time: use Android Baseline Profiles (AGP 7.3+) to pre-compile critical paths. Ask the AI to generate the baseline profile instrumentation test.

4.5  A/B Test Ideas (Privacy-Safe)
All A/B tests must respect the privacy-first design. No external analytics SDK required. Use local feature flags.
    • A/B test: Default WPM on first launch (200 vs 250). Measure: does user change it within first session? Log locally only.
    • A/B test: Upgrade prompt copy variant A vs B. Measure: conversion rate (local DataStore counter). No PII.
    • A/B test: Streak notification timing (morning vs evening). Measure: session start rate next day. Local only.

4.6  Quality Gates (Always Apply)
Before any AI-generated change is committed to develop:

  [ ] ./gradlew test passes (unit tests).
  [ ] ./gradlew connectedAndroidTest passes (instrumented tests, if applicable).
  [ ] Human developer has read every changed line.
  [ ] No hardcoded keys, secrets, or product IDs introduced.
  [ ] No new permissions added without explicit justification.
  [ ] CHANGELOG.md updated with a one-line description of the change.
